Separate those Concerns


  • Application setup is a concern
  • A running app doesn't need to know about setup
  • Avoid ad-hoc initialization 
  • Don't scatter setup across the system 
  • Find consistent strategy for resolving dependencies


WHY?


  • Avoids setup logic intermixing with runtime logic
  • Testability
  • Flexibility


How?


  • Wire everything in "main"
  • Factories
    • control when objects are created
  • Dependency Injection

Dependency Injection


  • Dedicated objects for secondary responsibilities
  • Classes don't resolve their own dependencies
  • Actual objects used specified by a config 




Scaling


software systems can grow incrementally

The Cautionary Tale of EJB

  • Concerns separated in some areas
  • Security, persistence, and transaction strategies declared in deployment descriptors
  • HOWEVER
  • Business logic coupled to the app container
  • Testing very difficult
  • Abstractions offer little reus e

Cross cutting concerns

 
  • Individual components (e.g. persistence components and domain logic) may be modular but they intersect
  • Implementation is required across many objects
  • Would rather delegate to a framework where behavior modification is made noninvasively .
